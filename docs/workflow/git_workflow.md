# Регламент работы с git, ветками и базой данных (СУБД)

## 1. Ветки git

- `dev` – основная ветка разработки. Сюда сливаются все функциональные ветки через Pull Request (PR).
- `test` – ветка для автоматического тестирования (unit-тесты). Сюда попадает код из `dev` после завершения разработки прецедента.
- `stage` – предпродакшн. Сюда попадает код из `test` после успешного прохождения unit-тестов. Здесь проводится ручное тестирование.
- `main` (`prod`) – продакшн. Сюда попадает код из `stage` после ручного тестирования.

## 2. Правила работы с git (разработка кода)

### Разработка новой фичи (прецедента)
1. Разработчик переключается на свежую ветку `dev`:  
   `git checkout dev`  
   `git pull origin dev`
2. Создаёт свою feature-ветку от `dev`:  
   `git checkout -b feature/название-задачи` (например, `feature/order-checkout`)
3. Пишет код, коммитит, пушит:  
   `git push -u origin feature/название-задачи`
4. На GitHub создаёт Pull Request (PR) из своей ветки в `dev`.
5. Тим-лид или архитектор проверяет PR, одобряет, затем сливает.

### Перемещение кода по конвейеру
- **Из `dev` в `test`** – только ответственный за git. Когда задача полностью готова и все unit-тесты проходят (запущены локально), он делает:  
  `git checkout test`  
  `git merge dev`  
  `git push origin test`
- **Из `test` в `stage`** – после успешного прогона unit-тестов (автоматически или вручную тестировщиком).  
- **Из `stage` в `prod`** – раз в неделю (или после полного цикла тестирования) тим-лид делает merge.

### Ветка `prod` (main) обновляется только из `stage`.

## 4. Примеры команд git

```bash
# Создать feature-ветку
git checkout dev
git pull
git checkout -b feature/add-product

# Коммит с сообщением
git add .
git commit -m "Добавлен эндпоинт для создания товара"

# Пуш и создание PR
git push -u origin feature/add-product
# (далее на GitHub создать PR)

# Слияние dev -> test (делает ответственный за git)
git checkout test
git pull origin test
git merge dev
git push origin test
```

## 5. Работа с базой данных (СУБД) в разных ветках

### 5.1. Общие принципы
- Схема базы данных (таблицы, индексы, ограничения) описывается **миграциями** (скриптами изменения схемы). Миграции хранятся в папке `backend/migrations/` и являются частью кода.
- Для работы с миграциями используется инструмент **Alembic** (для FastAPI + SQLAlchemy) или аналогичный.
- Каждая миграция получает уникальный идентификатор (например, `20250503_01_create_orders_table.py`).
- Все изменения схемы (добавление/удаление таблиц, колонок, индексов) фиксируются только через миграции, **никаких ручных правок в БД**.

### 5.2. Базы данных для каждой ветки
Для изоляции разных этапов разработки используются **отдельные экземпляры баз данных** (или отдельные схемы в одной PostgreSQL). Рекомендуется:

| Ветка   | База данных                     | Наполнение данных                                    |
|---------|--------------------------------|------------------------------------------------------|
| `dev`   | `shop_dev`                     | Минимальные тестовые данные (фикстуры) для разработки |
| `test`  | `shop_test`                    | Данные для unit-тестов (чистая схема + фикстуры)      |
| `stage` | `shop_stage`                   | Копия продакшен‑данных (анонимизированная) + тестовые |
| `prod`  | `shop_prod`                    | Реальные данные (учебного проекта)                   |

### 5.3. Процесс изменения схемы БД при разработке фичи
1. Разработчик создаёт feature-ветку от `dev`.
2. При необходимости изменить схему (новая таблица, колонка) он **локально** создаёт новую миграцию:
   - Вносит изменения в модели SQLAlchemy.
   - Генерирует миграцию: `alembic revision --autogenerate -m "описание"`.
   - Редактирует скрипт миграции при необходимости.
   - Применяет миграцию к своей локальной БД `shop_dev`: `alembic upgrade head`.
3. Коммитит **и код, и файл миграции** в свою feature-ветку.
4. После слияния PR в `dev` ответственный за git (или разработчик СУБД) применяет миграции к общей БД `shop_dev` (на тестовом сервере или локально командой).

### 5.4. Продвижение миграций по конвейеру (dev → test → stage → prod)
- **При слиянии `dev` → `test`** (делает ответственный за git):
  - Автоматически или вручную запускаются миграции на БД `shop_test`:  
    `alembic upgrade head`
  - После этого выполняются unit-тесты (они используют `shop_test`).
- **При слиянии `test` → `stage`**:
  - Миграции применяются к БД `shop_stage` (`alembic upgrade head`).
  - Производится копирование (обновление) данных из `prod` (если нужно) с осторожностью, чтобы не потерять данные.
- **При слиянии `stage` → `prod`**:
  - **Перед слиянием** обязательно создаётся резервная копия `shop_prod`.
  - Затем применяются миграции к `shop_prod` (`alembic upgrade head`).
  - После успешного обновления схемы обновляется код приложения (сервер).

### 5.5. Откат миграций (если что-то пошло не так)
- Если на любой ветке (кроме `prod`) миграция вызвала ошибку, можно откатиться на шаг назад:  
  `alembic downgrade -1`
- На `prod` откат требует отдельного решения тим-лида и архитектора. Обычно выполняется через `alembic downgrade` с последующим восстановлением резервной копии.

### 5.6. Синхронизация данных между ветками (stage и prod)
Раз в неделю (при обновлении `stage` из `test`) выполняется синхронизация данных из `prod` в `stage` (или наоборот – по заданию: затираете содержимое базы на stage из prod). Процесс:
1. Сделать дамп БД `shop_prod` (без конфиденциальных данных, если есть).
2. Восстановить дамп в `shop_stage` (предварительно очистив её).
3. Применить к `shop_stage` новые миграции (если есть отличия от `prod`).

Это гарантирует, что ручное тестирование на `stage` происходит на данных, максимально похожих на реальные.
